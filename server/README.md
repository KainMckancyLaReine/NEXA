# NEXA — Backend (local execution engine)

A zero-dependency Node server that turns the NEXA front-end into a working
product: persistent workforce, a live activity stream, and an **agent
execution engine** that carries out commands as action chains.

## Run

```bash
node server/server.js
```

Then open **http://localhost:4000** — the server hosts the whole site *and*
the API, so the Command Center, Marketplace, and employee pages all run live.
No `npm install` required (Node 16+).

Reset demo data: `curl -X POST http://localhost:4000/api/reset`

## What's real vs. sandbox

| Layer | Status |
|---|---|
| Persistence (`server/data/nexa.json`) | **Real** — survives restarts |
| Workforce / install / metrics | **Real** — state mutates and persists |
| Activity stream & polling | **Real** — generated by the engine each tick |
| Command routing + executed chains | **Real** — `engine.js` |
| Gmail / CRM / ERP / Calendar connectors | **Sandbox** — simulated, no real accounts touched |

The connectors in `connectors.js` deliberately **never** send email, move money,
or mutate an external system. They return realistic fabricated results so the
engine can demonstrate end-to-end execution safely.

## Making the connectors real

1. Add OAuth credentials per provider as environment variables.
2. Replace each `simulate*`/sandbox body in `connectors.js` with a real API call
   (e.g. Gmail API, HubSpot, Exact).
3. Gate every side-effecting action behind an approval policy before it runs.

## API

| Method | Endpoint | Purpose |
|---|---|---|
| GET | `/api/state` | company, connectors, workforce |
| GET | `/api/workforce` | installed AI employees + metrics |
| GET | `/api/employee/:id` | one employee + its action history |
| GET | `/api/activity?since=ts` | activity feed (polling) |
| GET | `/api/intelligence` | executive intelligence panel data |
| GET | `/api/marketplace` | installable AI employees |
| POST | `/api/workforce/install` | `{ id }` — hire an employee |
| POST | `/api/commands` | `{ text }` — dispatch a command, returns executed chain |
| POST | `/api/reset` | reseed demo data |

## Files

```
server/
  server.js      HTTP server: REST API + static hosting
  engine.js      NexaEngine — tick loop + command routing + action chains
  connectors.js  Sandbox connectors (swap for real APIs)
  store.js       JSON persistence + employee library + seed
  data/nexa.json created on first run
```

> Note: opening the HTML files directly (file://) still works — the front-end
> falls back to local demo data when the API isn't reachable.
